# YOLO - HGCAL

## Simulation

**Main repository:** https://github.com/sidhu2690/GEANT4/tree/main/HGCAL

The HGCAL workflow is organised into three main directories:

```
HGCAL/
├── Codes/
├── Pileup_Simulation/
└── Single_Particle_Simulation/
```

| Directory | Description |
|---|---|
| `Single_Particle_Simulation/` | Used for generating signal events (electrons, photons, positrons, etc.) |
| `Pileup_Simulation/` | Used for generating pileup/background events |
| `Codes/` | Contains utilities used in later stages of the workflow, such as segmentation, image creation, and analysis |

---

## Step 1: Signal Event Generation

**Directory:** https://github.com/sidhu2690/GEANT4/tree/main/HGCAL/Single_Particle_Simulation

This directory contains the Geant4-based simulation used to generate signal events.

### Configure the Particle

Open `generator.cc` and modify the generator configuration according to the sample you want to generate:

- Particle ID
- η (eta)
- φ (phi)
- pT
- Any other required generator parameters

### Configure the Run

Navigate to `Single_Particle_Simulation/build/` and edit `run.mac` to set the required number of events. Also update the seed value if needed.

### Running the Simulation

```bash
qsub run.sh
```

Make sure Geant4 is installed and the environment is properly configured before running the simulation.

### Output

A successful run produces a Step-1 ROOT file similar to:

```
Electron_nPU_000_Pt_025_Eta_170_Events_2K_Set01_Step1.root
```

These are referred to as **Step-1 files** and contain the detector SimHits generated by Geant4.

---

## Step 2: Cell-wise Segmentation

**Directory:** https://github.com/sidhu2690/GEANT4/tree/main/HGCAL/Codes/Segmentation

This stage converts SimHits from the Step-1 file into detector-cell level DigiHits.

```
Segmentation/
├── automation.C
├── cellwise_segmentation.C
└── job.sh
```

| File | Description |
|---|---|
| `cellwise_segmentation.C` | Main segmentation macro used to convert SimHits into DigiHits |
| `automation.C` | Used for batch processing and parallel execution when a large number of Step-1 ROOT files need to be processed |
| `job.sh` | PBS submission script used for running segmentation jobs on a cluster |

### Input

```
Electron_nPU_000_Pt_025_Eta_170_Events_2K_Set01_Step1.root
```

### Output

```
Electron_nPU_000_Pt_025_Eta_170_Events_2K_Set01_Step2.root
```

These are referred to as **Step-2 files**. They contain the digitised detector response (DigiHits) after cell-wise segmentation and serve as input to the next stages of the workflow.

---

## Step 3: Pileup Event Generation

**Directory:** https://github.com/sidhu2690/GEANT4/tree/main/HGCAL/Pileup_Simulation

For pileup generation, a Pythia-generated ROOT file is required as input. Before running the pileup simulation, convert the Pythia ROOT file into a text file using:

https://github.com/sidhu2690/GEANT4/blob/main/HGCAL/Codes/PileupUtils/01_rootToText.C

### ROOT to Text Conversion

1. Open `01_rootToText.C`
2. Modify the code if required according to the structure of your Pythia ROOT file
3. Specify the path to the Pythia ROOT file inside the macro
4. Run the ROOT macro

**Output:** `generated_data.txt`

### Running the Pileup Simulation

1. Go back to https://github.com/sidhu2690/GEANT4/tree/main/HGCAL/Pileup_Simulation
2. Copy `generated_data.txt` into the build directory
3. If needed, modify the output ROOT file name in `run.cc`
4. You can also modify `run.mac` to change the number of events. Similarly, update the seed value if required.
5. Run the simulation:

```bash
qsub run.sh
```

**Output:** `PileUp_Pt_GT_pt3_Eta_15_31_Events_20K_Step1.root`

This is the pileup Step-1 file.

### Cell-wise Segmentation

The pileup Step-1 file can be processed through the same segmentation workflow described earlier:

https://github.com/sidhu2690/GEANT4/tree/main/HGCAL/Codes/Segmentation

**Input:** `PileUp_Pt_GT_pt3_Eta_15_31_Events_20K_Step1.root`

**Output:** `PileUp_Pt_GT_pt3_Eta_15_31_Events_20K_Step2.root`

## Step 4: Adding Pileup to Signal

Once the signal and pileup files have been digitized, they can be combined.

**Required files:**
- `Electron_nPU_000_Pt_025_Eta_170_Events_2K_Set01_Step2.root`
- `PileUp_Pt_GT_pt3_Eta_15_31_Events_20K_Step2.root`

**Use:** https://github.com/sidhu2690/GEANT4/blob/main/HGCAL/Codes/SignalToPileupAdittion/addPileupToSignal_03.C

Inside the macro, specify:

```cpp
void addPileupToSignal(
    const char* signalFile = "Electron_Pt_025_Eta_170_Events_2K_PU_000_Set01_Step2.root",
    const char* pileupFile = "PileUp_Pt_GT_pt3_Eta_15_31_Events_20K_Step2.root",
    const char* outputFile = "Electron_nPU_35_Pt_025_Eta_170_Events_2K_Set01_Step2.root",
    int nPU = 35,
    int classLabel = 0)
```

| Parameter | Description |
|---|---|
| `signalFile` | Signal Step-2 file |
| `pileupFile` | Pileup Step-2 file |
| `outputFile` | Output file name |
| `nPU` | Number of pileup interactions |
| `classLabel` | Class identifier (for example, different values can be used for electrons and muons) |

**Directory:** https://github.com/sidhu2690/GEANT4/tree/main/HGCAL/Codes/SignalToPileupAdittion

Additional files `electron_var_pileup_automation.C` and `job.sh` can be used to automate and submit jobs.

**Example output:**

```
Electron_nPU_070_Pt_025_Eta_170_Events_2K_Set01_Step2.root
```

This non-zero pileup Step-2 file will be used later for the ML tasks.

## Step 5: Preparing Data for ML

The combined Step-2 ROOT files are converted into the format required for the ML task using:

https://github.com/sidhu2690/GEANT4/blob/main/HGCAL/Codes/YOLO/process_root.py

This script takes the combined ROOT file as input and produces the data required for training and testing.

### Training Data Preparation

**Use:** https://github.com/sidhu2690/GEANT4/blob/main/HGCAL/Codes/YOLO/submit_training.sh

Before running, make the necessary changes to the paths:

```bash
JOBDIR="/persistent/data1/ms21080/Regression/Data/electron"
INDIR="/persistent/data1/ms21080/Data/Signal_PileUp_Data/Electron_Var_Pileup"
OUTBASE="/persistent/data1/ms21080/Regression/Data/Electron/100"
```

Also configure the processing options inside the script:

```python
process_root_file(
    in_file=IN_FILE,
    prefix="ele_var_set${SET}",
    start_index=0,
    n_events=-1,
    val_fraction=0.0,
    test_fraction=0.0,
    box_size='l',
    base_dir=OUT_DIR,
    fixed_box=True,
    seed=41
)
```

Box size options: `'l'` → Large, `'n'` → Nominal, `'s'` → Small

Run the script after making the required modifications.

**Output directories used for training:**
```
images/
labels/
energy/
```

### Testing Data Preparation

**Use:** https://github.com/sidhu2690/GEANT4/blob/main/HGCAL/Codes/YOLO/submit_testing.sh

This script is used to generate testing data containing fixed η-pT event samples. Make the required changes to the file paths and processing configuration.

```python
process_root_file(
    in_file=IN_FILE,
    prefix="ele_PU${PU}_PT${PT}_ETA${ETA}",
    start_index=0,
    n_events=-1,
    val_fraction=0.0,
    test_fraction=0.0,
    box_size='l',
    base_dir=OUT_DIR,
    fixed_box=True,
    seed=42
)
```

Box size options: `'l'` → Large, `'n'` → Nominal, `'s'` → Small

After running the scripts, the data required for the ML task will be available.

## Step 6: YOLO Training Setup

**Repository:** https://github.com/Sid2690/HGCAL_YOLOV5_Linear

This repository contains the customized YOLOv5 implementation used for the HGCAL studies.

The three directories generated in the previous step (`images/`, `labels/`, `energy/`) should be copied into:

https://github.com/Sid2690/HGCAL_YOLOV5_Linear/tree/main/data

### Dataset Configuration

Inside https://github.com/Sid2690/HGCAL_YOLOV5_Linear/tree/main/data there are multiple configuration files such as `muon47m.yaml` and `hyps/`. The dataset yaml file can be modified if required.

Example:

```yaml
path: ./data
train: images/train
val: images/val

nc: 1
names: ['positron']
ch: 16
```

| Parameter | Description |
|---|---|
| `nc` | Number of classes |
| `names` | Class names |
| `ch` | Number of input channels |

The value of `ch` must match the number of channels present in the prepared dataset.

### Hyperparameter Configuration

**File:** https://github.com/Sid2690/HGCAL_YOLOV5_Linear/blob/main/data/hyps/hyp_muon47.yaml

The main parameters that are typically modified are:

```yaml
lr0: 0.001
lrf: 0.01
momentum: 0.937
weight_decay: 0.0005
warmup_epochs: 5.0
warmup_momentum: 0.8
warmup_bias_lr: 0.1
box: 0.05
cls: 0.5
energy: 0.1
cls_pw: 2.0
obj: 1.0
obj_pw: 1.0
iou_t: 0.2
anchor_t: 4.0
```

The remaining parameters in the file correspond to augmentations and are generally not modified since they may distort the physics information.

### Model Configuration

**File:** https://github.com/Sid2690/HGCAL_YOLOV5_Linear/blob/main/models/yolo47m.yaml

Example:

```yaml
# Parameters
nc: 1
ch: 16
depth_multiple: 0.67
width_multiple: 0.75

anchors:
  - [3,3, 5,5, 8,10]
  - [10,13, 16,30, 33,23]
  - [30,61, 62,45, 59,119]
```

The following parameters can be modified: `nc`, `ch`, `depth_multiple`, `width_multiple`, `anchors`. Architectural modifications can also be made through this file if required.

### Important Note

The `HGCAL_YOLOV5_Linear` repository is configured for 16-channel input data. If the number of channels is changed from 16, the setup may break unless the corresponding code and data are modified consistently.

For experiments involving different channel configurations, use:

https://github.com/Sid2690/HGCAL_YOLOV5_Custom

This repository currently uses 17-channel input data, and channel-related experiments can be performed through the yaml configuration without breaking the setup. Apart from that, the workflow remains the same.

## Step 7: Training and Testing on HPC

Before training, it is recommended to become familiar with GPU usage on the IISER HPC system:

https://iisermhpc.pages.dev/gpu_.html

### Installing TMUX

TMUX is required so that training sessions continue running even if the terminal connection is lost.

**Install libevent:**

```bash
cd ~
wget https://github.com/libevent/libevent/releases/download/release-2.1.12-stable/libevent-2.1.12-stable.tar.gz
tar -xzf libevent-2.1.12-stable.tar.gz
cd ~/libevent-2.1.12-stable
./configure --prefix=$HOME/local
make -j4
make install
```

**Install TMUX:**

```bash
cd ~
wget https://github.com/tmux/tmux/releases/download/3.4/tmux-3.4.tar.gz
tar -xzf tmux-3.4.tar.gz
cd tmux-3.4
CPPFLAGS="-I$HOME/local/include" \
LDFLAGS="-L$HOME/local/lib" \
PKG_CONFIG_PATH="$HOME/local/lib/pkgconfig" \
./configure --prefix=$HOME/local
make -j4
make install
```

**Add library path:**

```bash
echo 'export LD_LIBRARY_PATH=$HOME/local/lib:$LD_LIBRARY_PATH' >> ~/.bashrc
source ~/.bashrc
```

**Check installation:**

```bash
$HOME/local/bin/tmux -V
```

Expected output: `tmux 3.4`

**Add tmux to PATH:**

```bash
echo 'export PATH=$HOME/local/bin:$PATH' >> ~/.bashrc
source ~/.bashrc
```

**Test:**

```bash
tmux new -s test
```

### Training

Go to the model directory and activate the environment:

```bash
cd /persistent/data1/ms21080/Regression/Data/Model_Nominal
conda activate yolov5
unset LD_LIBRARY_PATH
export LD_LIBRARY_PATH=/usr/local/cuda-11.6/targets/x86_64-linux/lib:/usr/local/cuda-11.6/lib64
```

#### Creating Cache Files

Before running DDP training, first run a normal training job once so that the dataset cache files are created:

```bash
python train.py \
    --data data/muon47m.yaml \
    --cfg models/yolo47m.yaml \
    --hyp data/hyps/hyp_muon47.yaml \
    --weights '' \
    --img 736 \
    --batch-size 8 \
    --epochs 30 \
    --device 1 \
    --workers 2
```

This step is only used to load the data and generate the cache files. Once the cache files have been created, the process can be terminated and DDP training can be started.

#### DDP Training

```bash
export NCCL_TIMEOUT=1800
export TORCH_NCCL_BLOCKING_WAIT=1
export NCCL_DEBUG=INFO

python -m torch.distributed.run --nproc_per_node 4 train.py \
    --data data/muon47m.yaml \
    --cfg models/yolo47m.yaml \
    --hyp data/hyps/hyp_muon47.yaml \
    --weights '' \
    --img 736 \
    --batch-size 8 \
    --epochs 100 \
    --device 0,1,2,3 \
    --workers 2 \
    --optimizer Adam \
    --cos-lr \
    --project /persistent/data1/ms21080/Regression/Data/Model_Nominal/runs/train/ \
    --name v6
```

The model directory path and project/output path should be modified according to your setup.

| Argument | Description |
|---|---|
| `--data` | Dataset configuration file |
| `--cfg` | Model architecture configuration |
| `--hyp` | Hyperparameter configuration file |
| `--img` | Image size |
| `--batch-size` | Batch size used during training |
| `--epochs` | Number of training epochs |
| `--device` | GPU devices used for training |
| `--optimizer Adam` | Use Adam optimizer |
| `--cos-lr` | Enable cosine learning-rate scheduling |
| `--project` | Directory where training outputs will be stored |
| `--name` | Run name |

### Testing

```bash
cd /persistent/data1/ms21080/Regression/Data/Model_Nominal
conda activate yolov5
unset LD_LIBRARY_PATH
export LD_LIBRARY_PATH=/usr/local/cuda-11.6/targets/x86_64-linux/lib:/usr/local/cuda-11.6/lib64

python detect.py \
    --weights '/persistent/data1/ms21080/Regression/Data/Model/runs/train/v1/weights/best.pt' \
    --source '/persistent/data1/ms21080/Regression/Data/Electron/Testing/PT025/images/train' \
    --data data/muon47m.yaml \
    --img 736 \
    --conf-thres 0.4 \
    --iou-thres 0.25 \
    --device 3 \
    --save-txt \
    --save-conf \
    --project /persistent/data1/ms21080/Regression/Data/Electron/Testing/PT025/labels \
    --name v1
```

The model weights path, testing data path, and output project path should be updated according to your setup.

| Argument | Description |
|---|---|
| `--weights` | Path to the trained model |
| `--source` | Input testing dataset |
| `--conf-thres` | Confidence threshold |
| `--iou-thres` | IoU threshold |
| `--device` | GPU used for inference |
| `--save-txt` | Save predictions as text files |
| `--save-conf` | Save prediction confidence values |
| `--project` | Output directory for predictions |
